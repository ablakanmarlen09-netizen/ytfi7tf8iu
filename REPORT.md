## 3. Анализ требований

### 3.1 Функциональные требования

**Определение:** Функциональные требования описывают **что именно** система должна делать - конкретные функции, операции и поведение.

**Примеры из проекта:**
- "Пользователь может зарегистрироваться" - это действие, которое должна поддерживать система
- "Администратор может добавлять книги" - конкретная функция системы

**Визуализация:** Все функциональные требования полностью визуализированы через интерактивные экраны и формы.

### 3.2 Нефункциональные требования

**Определение:** Нефункциональные требования описывают **как** система должна работать - качественные характеристики и ограничения.

**Категории нефункциональных требований в проекте:**

1. **Usability (Удобство использования)**
   - Требование: "Интерфейс должен быть удобным и понятным"
   - Реализация: Четкая навигация, визуальная обратная связь, понятные метки

2. **Compatibility (Совместимость)**
   - Требование: "Корректное отображение на мобильных устройствах"
   - Реализация: Адаптивная верстка (responsive design)

3. **Performance (Производительность)**
   - Требование: "Время отклика поиска < 2 секунд"
   - Реализация: Оптимизация алгоритмов, мемоизация

4. **Security (Безопасность)**
   - Требование: "Базовая защита данных"
   - Реализация: Валидация, комментарии о шифровании

**Визуализация:** Частично визуализированы, частично описаны в комментариях и аннотациях.

#### ✅ Полностью визуализированные требования:

**Функциональные (100%):**
- Все 8 функциональных требований визуализированы через статичные макеты экранов
- Каждая функция представлена отдельным экраном с UI-элементами
- Создано 11 экранов, покрывающих все пользовательские сценарии

**Нефункциональные (визуализация + аннотации):**
- **Удобство интерфейса** - визуализировано через интуитивный дизайн, навигацию, цветовые схемы
- **Мобильная адаптивность** - визуализировано через адаптивную верстку Tailwind CSS
- **Производительность поиска** - показана аннотация на экране каталога с требованием < 2 сек
- **Защита данных** - аннотации на экранах Login, Register, Checkout

#### 📝 Что осталось в виде описаний (аннотаций):

**Нефункциональные требования, невидимые для пользователя:**
- Шифрование трафика (HTTPS) - описано в желтых блоках-аннотациях
- Хеширование паролей (bcrypt) - описано в аннотациях форм
- Защита от атак (CSRF, XSS) - описано в документации
- Мониторинг производительности - описано в аннотации каталога

**Причина:** Эти требования не могут быть **визуально показаны** в прототипе, так как:
1. Работают на серверной стороне (шифрование, хеширование)
2. Не видны пользователю (внутренние процессы)
3. Требуют реальной инфраструктуры для демонстрации

**Решение:** Размещены аннотации в желтых блоках на соответствующих экранах для документирования этих требований.

## 5. Технические детали реализации

### 5.1 Архитектура прототипа

**Тип:** Нефункциональный прототип (статичные макеты) на базе React SPA

**Структура навигации:**
```
App (RouterProvider)
  └── Layout (Header + Footer + Outlet)
       ├── Home (главная с баннером)
       ├── Catalog (макет каталога с поиском)
       ├── BookDetails (детали книги)
       ├── Cart (корзина со статичными товарами)
       ├── Checkout (форма заказа)
       ├── Login (форма входа)
       ├── Register (форма регистрации)
       └── Admin
            ├── Dashboard (статистика)
            ├── Add Book (форма добавления)
            └── Orders (управление заказами)
```

**Особенность:** Все формы и кнопки отключены (disabled) - это только макеты для демонстрации UI.

### 5.2 Статичные данные

**AppContext содержит только мок-данные:**
- `mockBooks[]` - 6 демонстрационных книг
- `mockCart[]` - 2 товара для визуализации корзины
- `mockOrders[]` - 2 заказа для админ-панели
- `mockUser` - null (пользователь не авторизован)

**Никаких функций:** Все методы удалены, Context предоставляет только статичные данные.

### 5.3 Визуализация нефункциональных требований

**Производительность поиска:**
- На странице Catalog размещена аннотация в желтом блоке
- Текст: "Время отклика при поиске не должно превышать 2 секунд"
- Пояснение: "В рабочем прототипе: useMemo + консольный мониторинг"

**Мобильная адаптивность:**
- Реализована через Tailwind CSS responsive utilities
- Grid меняется: 1 колонка (mobile) → 2 (tablet) → 3-4 (desktop)
- Можно проверить через DevTools responsive mode

**Безопасность:**
- Аннотации на страницах Login, Register, Checkout
- Описаны требования: HTTPS, bcrypt, JWT, валидация
- Показаны требования к паролю (минимум 6 символов)
- Демо-аккаунты для тестирования макетов

### 5.4 Отличия от функционального прототипа

| Аспект | Нефункциональный | Функциональный |
|--------|------------------|----------------|
| Кнопки | Disabled, не работают | Кликабельные, выполняют действия |
| Формы | Отключены, только дизайн | Валидация, отправка данных |
| Данные | Статичные моки | Динамическое состояние (useState) |
| Методы | Отсутствуют | addToCart(), login(), etc. |
| Цель | Показать UI и требования | Реальная работа приложения |

### 7.3 Что нельзя полностью визуализировать в нефункциональном прототипе

**Технические НФТ (только аннотации):**
- ❌ Реальное HTTPS-шифрование (работает на сервере)
- ❌ Хеширование паролей (внутренний процесс)
- ❌ Измерение реального времени поиска (нужна функциональность)
- ❌ Защита от CSRF/XSS атак (серверная логика)

**Решение:** Эти требования:
1. Описаны в аннотациях на экранах (желтые блоки)
2. Задокументированы в REQUIREMENTS.md
3. Будут реализованы при создании функционального прототипа/production

### 7.4 Преимущества нефункционального прототипа

✅ **Быстрое создание** - не нужно писать логику
✅ **Фокус на UI/UX** - концентрация на дизайне
✅ **Простое тестирование** - показать заказчику макеты
✅ **Раннее выявление проблем** - UX-тестирование до разработки
✅ **Наглядная документация** - аннотации прямо на экранах

### 7.5 Когда переходить к функциональному прототипу

После утверждения:
- Структуры экранов
- Навигации между страницами
- Визуального дизайна
- Расположения элементов

Можно добавлять:
- Реальную логику (useState, обработчики)
- Валидацию форм
- Интеграцию с API
- Измерение производительности
